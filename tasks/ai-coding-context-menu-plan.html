后台服务做重活

统一 Settings

会话自动桥接

用户只选择用哪个 agent 打开；同渠道直接 resume，跨渠道由后台自动判断和接续上下文。

右键菜单规划

右键菜单采用 TortoiseGit 式分层，但默认一级菜单只保留 4 项：Claude Code、Codex、Gemini、CC-Menu。 其他动作都进入 CC-Menu 二级菜单。Desktop 账号实时切换不做三级菜单，而是按 Agent 分组直接展开到二级菜单，并用分割线拆分。 启动路径有硬边界：一级菜单的 Claude/Codex/Gemini 只做原生默认启动；CC-Menu 二级菜单里的 Start with Options 和账号渠道项通过 Gateway 启动或切换。 Windows 上接入 Explorer context menu，macOS 上接入 Finder 的 Services/Quick Actions/Finder Sync Extension。 菜单项由 Settings -> Agents 的 agent 配置动态生成；新增、修改、删除 agent 后，后台服务重新生成菜单缓存。

一级菜单

  • Open Claude Code Here：在当前目录用 Claude Code 默认配置启动，不走 Gateway。
  • Open Codex Here：在当前目录用 Codex 默认配置启动，不走 Gateway。
  • Open Gemini CLI Here：在当前目录用 Gemini CLI 默认配置启动，不走 Gateway。
  • CC-Menu：进入启动参数、会话、cc-switch Agent 同步和统一 Settings。

CC-Menu 二级菜单

  • Start with Options：打开 Gateway 启动参数对话框，选择 agent、模型渠道、缓存、TTS 等参数，并注入本地 Gateway 配置。
  • Sessions：打开独立 Sessions 界面，按当前目录过滤会话。
  • Sync Agents from CC-Switch：从 cc-switch 同步 agent/profile 信息，合并到 Agents tab。
  • Sync Menus：按当前已配置的大模型渠道账号重新生成右键菜单缓存，并刷新 Explorer/Finder 菜单适配层。
  • Settings：打开统一 Settings，配置 Agents、Context Menu、会话扫描、Gateway 和其他设置。
  • Claude Code accounts：SSSAi、官方、Work 等配置直接显示在二级菜单，点击即切换 Gateway route，并让 Claude Code Desktop/CLI 新会话使用该 route。
  • Codex accounts：SSSAi、官方、Work 等配置直接显示在二级菜单，点击即切换 Gateway route，并让 Codex Desktop/CLI 新会话使用该 route。
  • Gemini accounts：Personal、Workspace 等配置直接显示在二级菜单，点击即切换当前 Gemini Gateway route。

约束：Explorer/Finder 菜单打开不能等待会话扫描、网络请求或 Agent 验证。 菜单数据来自后台服务写好的轻量缓存；agent 或账号配置变更后由 service 自动重建缓存，用户也可以点 Sync Menus 手动刷新。

Windows 菜单适配

  • 优先支持 Explorer 传统 context menu 和 Windows 11 compact menu。
  • Shell extension 只取路径、读缓存、调用 broker。
  • 启动 terminal 支持 Windows Terminal、PowerShell 7、cmd 和自定义模板。
  • 密钥优先进入 Windows Credential Manager/DPAPI。

macOS 菜单适配

  • 第一版 macOS 可用 Finder Quick Actions/Services 做低风险入口。
  • 增强版使用 Finder Sync Extension 提供更贴近右键菜单的体验。
  • 启动 terminal 支持 Terminal.app、iTerm2、Warp 和自定义命令。
  • 密钥进入 Keychain，后台服务用 launchd agent 管理。

Rust Gateway 设计

Gateway 不是完整复刻 LiteLLM，而是为本机 AI 编程工作流做一个轻量、高性能、可热切换的路由层。 实现目标一次性覆盖 OpenAI-compatible endpoint、Anthropic、Gemini、Local 和自定义 provider adapter，不按产品阶段拆交付。

路由策略

  • fixed：固定 provider 和 model。
  • fallback：失败或超时后切换到备用渠道。
  • race：多个 provider 并发，最快成功者返回。
  • broadcast：多个 agent 同时执行，用于审查或方案对比。

技术选型

  • Async runtime：Tokio。
  • HTTP server：Axum 或 Hyper。
  • HTTP client：Reqwest。
  • Config：Serde + TOML/JSON。
  • Local DB：SQLite。
  • Credential：Windows Credential Manager 或 DPAPI。

会话索引与自动接续流程

会话扫描由独立后台服务完成，类似 Everything 的轻量索引思路。 右键菜单只读缓存，按当前目录过滤会话。用户只看见 “Resume/Open with Agent”，兼容性判断和上下文桥接都在后台完成。

Session Indexer

  • Rust 后台服务监控 Claude、Codex、Gemini 等会话目录。
  • 默认每 5 分钟触发一次全量轻扫描。
  • 目录发生修改时通过 watcher 触发增量扫描，写入 SQLite 索引。
  • 记录 session ID、channel、project path、title、last active、compatibility。
  • 不可解析格式只记录路径、mtime、大小和渠道猜测。

Rust 扫描服务性能策略

  • 使用 notify 监听目录变化，Windows 走 ReadDirectoryChangesW，macOS 走 FSEvents。
  • 文件事件进入 bounded channel，按目录和 session id 做 300-800ms debounce，避免编辑器连续写入导致重复解析。
  • 5 分钟 timer 只做轻扫描：比较 mtime、size、inode/file id、hash hint，变化后才解析正文。
  • SQLite 使用 WAL，索引字段覆盖 workspace hash、agent、provider session id、last active、source path。
  • 解析任务限制并发，低优先级运行；右键菜单只读缓存，不等待扫描。
  • 扫描器必须可重建索引，数据库损坏时删除重建，不影响源会话文件。

实时切换策略

  • 第三方 CLI/Desktop 尽量连接稳定本地 endpoint：http://127.0.0.1:<port>/v1。
  • 切换 profile 时由 gateway 即时改路由，不频繁改写第三方静态配置。
  • 必须读本地配置文件的工具，启动前生成临时配置并注入。
  • 已运行 Desktop 无法热更新时标记 restart_required。

安全边界

  • API key/token 不写入普通配置文件和日志。
  • 所有 command template 参数数组化，不拼接未校验字符串。
  • SQLite 索引可重建，不能成为唯一数据源。
  • Shell extension 禁止网络请求和会话扫描。

/goal 一次性执行说明

这段是给后续新开的 CLI 直接执行用的任务书。执行目标不是做 demo，也不是中途停在规划或半成品； 要完成整套开发、自测、发布和安装包验证，并提交到 GitHub public 仓库。

完整实现范围

  • Rust workspace：service、gateway、session-indexer、agent-registry、menu-cache、launch-broker、platform adapters。
  • Windows Explorer 右键菜单：一级原生默认启动；CC-Menu 二级 Gateway 启动、账号切换、Sync Menus、Sessions、Settings。
  • macOS 预留并实现可用路径：Services/Quick Actions/Finder Sync adapter、launchd、Keychain、Terminal/iTerm2/Warp 模板。
  • Settings UI：Agents 卡片、hover 小按钮、参数弹窗、Context Menu、会话扫描、Gateway、Settings tab。
  • Sessions UI：当前目录过滤，宽二级树 Agent -> 会话，行右键菜单负责 Resume/Open with。
  • Rust Gateway：OpenAI-compatible endpoint、Anthropic/Gemini/Local/custom adapters、fixed/fallback/race/broadcast。
  • Rust 会话扫描服务：5 分钟默认轻扫描、文件 watcher 增量扫描、SQLite WAL、可重建索引。
  • cc-switch 同步：dry-run、冲突处理、secret 进 Credential Manager/DPAPI 或 Keychain。

不中断执行规则

  • 不要只生成文档或脚手架；必须完成可运行产品。
  • 不要把 Rust Gateway、Sessions、Installer 拆成后续任务；都要在同一个 /goal 内完成。
  • 遇到缺少第三方真实账号时，用 fixture provider 和 fake CLI 完成端到端测试，同时保留真实配置入口。
  • 所有平台相关能力必须有 adapter 抽象；Windows 先重点自测，macOS 路径至少完成编译/结构和文档化验证。
  • 提交前必须跑自动化测试、UI 冒烟测试、安装包安装/卸载测试和菜单缓存生成测试。
自测项 必须验证 通过标准
Rust 单元/集成测试 gateway routing、menu projection、session scan、config validation、credential refs。 cargo test --workspace 全绿。
菜单流程 一级菜单原生启动不注入 Gateway；二级菜单 Gateway 启动会注入 route。 fake CLI 捕获 cwd/env/args，与预期一致。
会话扫描 5 分钟 timer、watcher 事件、debounce、增量更新、SQLite 重建。 修改 fixture 会话文件后索引自动更新，右键菜单不阻塞。
Sessions UI 当前目录过滤、Agent -> 会话二级树、会话行右键菜单。 用 fixture 数据打开 UI，树和右键菜单均可操作。
Settings UI Agents 卡片 hover action、编辑弹窗、Sync Agents、Sync Menus。 保存配置后菜单缓存重建，错误配置有明确提示。
安装包 安装、启动 service、注册菜单、卸载清理。 全流程在干净测试目录自测通过，有日志和截图/文本记录。

安装程序要求

  • Windows 优先提供 NSIS/MSI 之一，安装 service、UI、shell extension/menu registration。
  • 安装前检测旧版本，支持升级；卸载时移除菜单注册、service、启动项，保留用户配置需询问或文档说明。
  • 安装包必须签名预留；没有证书时明确标记 unsigned build。
  • 安装包自测必须覆盖安装、重启 service、打开 Settings、Sync Menus、卸载。

可给新 CLI 的 /goal 指令：根据 tasks/ai-coding-context-menu-plan.html 完整实现 CC Menu。 不要中途停止在计划、脚手架或半成品；完成 Rust 应用、Windows/macOS 适配、Settings、Sessions、Gateway、会话扫描服务、安装程序、自测、GitHub public 仓库提交和 GitHub Release。 开发完成后必须运行完整测试矩阵，并验证安装包安装、菜单注册、service 启动、Sync Menus、会话扫描、卸载流程。

风险与开放问题

主要风险

  • Windows Shell extension 稳定性要求高，崩溃会影响 Explorer。
  • macOS Finder 扩展权限和沙盒限制更强，可能需要先用 Quick Actions/Services 降低风险。
  • Claude/Codex/Gemini 会话文件格式可能变化，parser 需要版本化。
  • Desktop 实时切换受第三方应用限制，部分场景只能保证新会话生效。
  • Agent 配置直接驱动菜单，如果校验不足，错误 command 会污染右键菜单或启动失败。
  • Gateway 容易膨胀成完整 LiteLLM 替代品，范围要锁定在本机 AI 编程启动、路由和会话接续。
  • cc-switch 配置格式可能变化，导入器必须先 dry-run。

开放问题

  • Windows 与 macOS 是否同一轮实现完整可用，还是 Windows 先落地、macOS 紧随。
  • 第一版是否只支持 Claude Code、Codex、Gemini CLI？
  • TTS 是播报结果，还是某个特定 CLI 参数/插件？
  • 默认 terminal 用 Windows Terminal、PowerShell 7，还是沿用 cc-tab？
  • macOS 默认 terminal 用 Terminal.app、iTerm2，还是 Warp？
  • cc-switch 配置路径和账号数据格式是否已有稳定约定？
  • cc-switch 同步是单向导入，还是需要双向同步？
  • 除 Claude/Codex/Gemini 外，自定义 Agent 是否允许提升到一级菜单，还是只在 Start with Options 中选择。
